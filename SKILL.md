# Carousel Universal System

## Назначение

Этот навык нужен для создания Instagram-каруселей почти на любую тему.

Он должен работать так:
1. определить тип задачи;
2. собрать нужную глубину intake;
3. при необходимости включить проверку данных;
4. при необходимости включить режим работы с фото;
5. собрать архитектуру, тексты, визуальную логику и QA.

## Главное правило

**Никогда не генерируй карусель сразу.**

Сначала всегда:
- определи тип задачи;
- собери intake;
- выяви недостающие параметры;
- при необходимости предложи варианты;
- явно зафиксируй defaults.

## Режимы

- `content-only` – обычная контентная карусель без обязательной проверки данных и без пользовательских фото;
- `data-first` – карусель с цифрами, сравнениями, доходностями, метриками или проверяемыми тезисами;
- `photo-based` – карусель, где пользовательские фото являются главным визуальным якорем;
- `mixed` – одновременно нужны и проверяемые данные, и пользовательские фото;
- `story` – история, путь, трансформация, кейс от лица героя;
- `sales` – боль, решение, оффер, CTA;
- `case-study` – контекст, проблема, действие, результат, вывод;
- `system` – пользователь просит не карусель, а саму систему, шаблон или проект.

## Decision protocol

### Как выбирать режим

- Если в задаче есть проверяемые цифры, сравнения, доходности, продуктовые claims или рыночные тезисы – включай `data-first`.
- Если пользователь просит использовать свои фото как основу визуала – включай `photo-based`.
- Если одновременно нужны и пользовательские фото, и реальные данные – включай `mixed`.
- Если пользователь просит историю, путь, личный опыт или нарратив – включай `story`.
- Если есть оффер, продажа, захват внимания к продукту или услуге – включай `sales`.
- Если пользователь просит структуру проекта, skill, prompt, шаблон или систему – включай `system`.
- Если нет ни обязательных данных, ни фото, ни специфического формата – используй `content-only`.

### Как выбирать глубину intake

- **Короткий intake**: если задача простая, без claims, без фото и без спорных данных.
- **Расширенный intake**: если есть фото, цифры, сравнения, продуктовые тезисы, доходности, оффер или несколько сценариев использования.
- **Остановка до уточнения**: если без ответа пользователя высок риск ошибиться в цели, аудитории, типе сравнения или роли фото.

## Intake

Собери минимум:
- тема;
- цель;
- аудитория;
- тип карусели;
- тон;
- главный акцент;
- есть ли свои факты, цифры, тезисы, офферы, примеры;
- нужен ли research;
- нужен ли персонаж / сюжет;
- нужны ли загруженные фото;
- нужен ли слой данных;
- какой нужен итог.

Если пользователь не знает ответ:
- не останавливайся;
- предложи 3–5 сильных вариантов по теме;
- коротко объясни разницу;
- если ответа всё равно нет, возьми разумный default и явно обозначь это.

## Проверка данных

Если в карусели есть:
- цифры;
- факты;
- сравнения;
- продуктовые характеристики;
- доходности;
- рыночные тезисы;
- метрики;
- claims,

то проверка обязательна.

### Severity protocol

- **Low risk**: общие описательные тезисы без чисел и без сильных обещаний.
- **Medium risk**: сравнения, продуктовые преимущества, рыночные оценки, тезисы вида `лучше`, `выгоднее`, `быстрее`.
- **High risk**: проценты, доходности, медицинские/финансовые claims, любые точные цифры, которые пользователь может воспринять как опору для решения.

### Verification rules

- Не выдумывай значения.
- Для medium/high risk claims по возможности сверяй больше одного источника.
- Если источники конфликтуют, используй диапазон или смягчай формулировку.
- Если надёжное подтверждение не найдено, не выноси цифру в headline как факт.
- Быстро стареющие данные помечай как актуальные на момент подготовки.
- В финальном пакете указывай, что нужно перепроверить перед публикацией.

Рабочий стандарт: **максимально проверенные и актуальные на момент подготовки данные**.

## Работа с фото

Если пользователь хочет использовать загруженные фото:
- считай фото главным визуальным якорем;
- сначала определи лицо, руки, ключевой объект и чистые зоны;
- не перекрывай лицо и эмоциональный центр кадра;
- определи `primary focus area`, `forbidden overlay area`, `safe text zone`, `secondary overlay area`;
- если фото слабое, честно рекомендуй `mixed` или чисто графический вариант.

## Правила текста

- 1 слайд = 1 мысль;
- коротко, но не обрубленно;
- без нейрослопа;
- без дешёвой драматизации;
- без пустых громких формул;
- без скрытых допущений;
- без выдуманных цифр.

## Anti-slop pass

Перед финальным выводом проверь:
- нет ли заголовков, которые звучат как generic-мотивация;
- нет ли фраз без проверяемого смысла;
- нет ли повторяющихся конструкций между слайдами;
- не спрятаны ли важные допущения;
- не звучит ли текст слишком шаблонно или “вирусно”; 
- можно ли сократить фразу без потери смысла.

Если текст гладкий, но пустой, перепиши самые слабые 2–3 слайда, а не полируй весь текст поверхностно.

## Визуальная логика

Для каждого слайда проектируй:
- scene;
- composition;
- key objects;
- mood;
- visual type;
- safe zones;
- overlays.

При необходимости используй:
- headline block;
- subheadline block;
- stat block;
- progress bar;
- badge / chip;
- footer strip;
- CTA band.

## Правило production

По умолчанию считай, что русский текст внутри изображения ненадёжен.

Режим production:
- сначала сцена;
- потом утверждённый текст отдельно;
- текст накладывается отдельным этапом.

## Режимы выдачи

### Compact
Используй, если пользователь просит краткий результат.

Структура:
1. Brief assumptions
2. Carousel architecture
3. Slide copy

### Standard
Используй по умолчанию для обычных задач без тяжёлого research.

Структура:
1. Brief assumptions
2. Carousel architecture
3. Slide copy
4. Visual system
5. QA notes

### Full production
Используй по умолчанию для `data-first`, `mixed`, сложных `photo-based` и production-задач.

Структура:
1. Brief assumptions
2. Carousel architecture
3. Slide copy
4. Visual system
5. Data and claims notes
6. Verification notes
7. Prompt pack
8. QA notes
9. System mode notes, если пользователь просил систему

## Если пользователь просит систему, а не карусель

Включай `system` и помогай собрать:
- instructions;
- стартовый prompt;
- output templates;
- чек-лист;
- reusable defaults.

## Короткая памятка

Порядок по умолчанию:
1. intake;
2. fallback options if needed;
3. research if needed;
4. verification if needed;
5. architecture;
6. slide copy;
7. anti-slop rewrite;
8. visual system;
9. prompts if needed;
10. QA;
11. final package.
